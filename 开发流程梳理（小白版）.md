## 你这个项目在做什么（用一句话）

你在做一个“像 Ant Design 那样的 React 组件库”，核心思路是：**先搭一套全局的设计系统（变量/重置/mixin/动画），再用这套系统去产出每个组件的 TSX + SCSS**。  
目前你已经把这条链路跑通到第一个组件：`Button`。

---

## 你按时间顺序做了哪些事（每一步都解释“为什么”）

下面按你描述的顺序，把每个文件的角色、它跟其它文件的关系、以及你在“开发流程”上实际完成了什么，完整串起来。

### 1) 先写 `src/styles/_variables.scss`：确定“设计变量”（Design Tokens）

你在 `src/styles/_variables.scss` 里做的事，本质是**定义组件库的“视觉规则”**，比如：

- **颜色体系**：`$primary/$danger/$gray-xxx` 等
- **字体体系**：`$font-size-base/$font-weight-base/$line-height-base` 等
- **全局基础变量**：`$spacer`（间距）、`$border-radius`（圆角）、`$box-shadow`（阴影）
- **组件相关变量**：例如你已经提前放了 Button 会用到的变量：`$btn-padding-*`、`$btn-border-radius-*`、`$btn-transition` 等

#### 为什么第一步先写它？

因为组件库要像 Ant Design 一样“统一风格”，你不希望每个组件都手写颜色/尺寸（那会非常乱、也难维护）。  
**变量文件就是你组件库的“统一标准”，以后改主题/改风格，大多只改这里。**

#### 你这一步的关键点

- 你大量使用了 `!default`：这是一种“可被覆盖的默认值”的写法（常见于 Bootstrap）。  
  意味着：将来如果你做主题覆盖（比如深色主题），可以在更前面定义同名变量来覆盖它。

---

### 2) 再写 `src/styles/index.scss`：做一个“全局样式入口”

你在 `src/styles/index.scss` 里做了一个很关键的工程化动作：**把所有全局样式按顺序集中引入**。

你现在的入口结构是：

- `@import "variables";`：先引入变量（后面文件会用到变量）
- `@import "reboot";`：再做全局重置（统一浏览器默认样式）
- `@import "mixin";`：再引入 mixin（给组件样式复用）
- `@import "animation";`：再放动画相关的 class/过渡
- `@import "../components/Button/style";`：再引入 Button 的组件样式

#### 为什么需要 `index.scss` 这种入口？

因为 React 项目最终只需要引入**一个**样式入口就能让全部样式生效。你把“加载顺序”和“引哪些模块”都放在这里统一管理，后续新增组件也只要往这里加一行 import。

#### 为什么 import 顺序很重要？

SCSS 里经常会出现“后引入覆盖前引入”。并且：

- `reboot` 会用到 `variables`（比如 body 字体、颜色）
- 组件样式会用到 `variables` + `mixin`

所以你现在这个顺序是合理的：**变量 → 全局基础 → 工具 → 动画 → 组件**。

---

### 3) 在 `src/App.tsx` 里改成 `import './styles/index.scss'`：把全局样式“接入应用”

这一句看起来很小，但它完成的是：**把你的组件库样式真正挂到页面里**。

你现在的 `App.tsx` 做了两件事：

- 引入 `./styles/index.scss`：让全局样式入口生效
- 引入并使用 `Button`：让你能在页面上验证 Button 的功能/样式

#### 为什么要在 App 里引入？

因为你现在还处在“组件库开发阶段”，需要一个地方做组件演示（类似一个 Playground / Demo）。  
`App.tsx` 就是你当前的“演示场”，你在里面写各种 `<Button ...>` 来验证：

- class 是否拼对了
- 禁用、大小、类型是否生效
- Link 类型是不是渲染成 `<a>` 而不是 `<button>`

---

### 4) 你又写了 `src/styles/_reboot.scss`：做全局 Reset / Reboot

`_reboot.scss` 的定位是：**把浏览器默认样式统一到一个可控的起点**（类似 Ant Design/Bootstrap 都会做的事）。

你做的核心点包括：

- `* { box-sizing: border-box; }`：统一盒模型，避免宽度计算混乱
- `body { margin: 0; font-family/font-size/line-height... }`：用你定义的变量给页面定“全局基础排版”
- `a`、`h1-h6`、`p`、`ul/ol`、`button/input` 等：把默认样式改成更一致、更适合组件库的基础表现

#### 为什么组件库需要 reboot？

如果不做 reboot，同一个组件在不同浏览器、不同页面默认 CSS 下可能表现不一致。  
组件库追求“开箱即用”和“稳定一致”，所以必须先把地基打平。

---

### 5) 你开始写第一个组件 `src/components/Button`：跑通“组件库最小闭环”

你在 `Button` 里完成了一个非常典型的组件库最小闭环（MVP）：

#### 5.1 `button.types.ts`：先定义“组件对外的 API”

你先把 Button 的可选项定义出来：

- `ButtonType`：`primary / default / danger / link`
- `ButtonSize`：`lg / sm`
- props：`disabled/size/btnType/href/children/className...`

这一步的意义是：**在写组件逻辑前，先把“用户怎么用它”规定好**。  
组件库最重要的不是内部怎么实现，而是对外 API 是否清晰、稳定。

#### 5.2 `button.tsx`：用 props 决定“渲染什么 + 挂哪些 class”

你在 `button.tsx` 里做的事是：

- 用 `classnames` 生成 class：
  - 永远有 `btn`
  - 根据类型加 `btn-${btnType}`（如 `btn-primary`）
  - 根据尺寸加 `btn-${size}`（如 `btn-lg`）
  - 当 Link + disabled 时，额外加 `disabled`（因为 `<a>` 没有原生 disabled）
- 根据类型决定渲染：
  - Link 且有 `href` → 渲染 `<a>`
  - 其它情况 → 渲染 `<button>`

这一步的意义是：**组件逻辑只负责“状态/结构/语义”，样式全部交给 class + SCSS。**

#### 5.3 `Button/_style.scss`：用变量 + mixin 产出“可复用的样式”

你的 Button 样式非常“组件库风格”：

- `.btn`：写基础外观（display、边框、阴影、过渡、disabled 行为）
- `.btn-lg / .btn-sm`：用 `@include button-size(...)` 生成不同尺寸
- `.btn-primary / .btn-danger / .btn-default`：用 `@include button-style(...)` 生成不同主题色的 hover/focus/disabled
- `.btn-link`：把链接按钮单独处理（去阴影、走链接颜色变量）

这一步的意义是：**把“重复的样式规律”抽到 mixin 里，组件样式只描述差异**，以后新增 `btn-warning` 这种类型就会很快。

## 这些文件是怎么串起来工作的（给小白的“调用链”）

你现在的链路可以用一句话记：

**`App.tsx` 引入 `styles/index.scss` → `index.scss` 先引入基础样式模块（variables/reboot/mixin/animation）→ 再引入组件样式（Button/style）→ 页面上渲染 `<Button>` 时挂上 class → SCSS 根据 class 生效。**

更直观一点（从“页面”到“样式”）：

- 页面写：`<Button btnType="primary" size="lg" />`
- 组件算出 class：`btn btn-primary btn-lg`
- SCSS 命中：
  - `.btn`（基础样式）
  - `.btn-primary`（主题色）
  - `.btn-lg`（尺寸）
- `.btn-primary` 又依赖：
  - 变量：`$primary/$white/...`
  - mixin：`button-style(...)`

## 你这个流程“像 Ant Design”的地方（你做对了什么）

- **先有设计变量，再写组件**：这是设计系统的标准做法
- **全局入口统一 import**：这让“新增组件样式”变成可维护的流程
- **组件 TSX 与 SCSS 分离**：逻辑与样式解耦，利于扩展
- **Link Button 语义正确**：Link 类型用 `<a>`，并处理 `<a>` 的 disabled（加 class + pointer-events）
- **mixin 抽象复用**：Button 的尺寸/颜色态统一由 mixin 输出，后续做其它组件可以复用同一套思路

## 小白下一步照着做（建议的后续开发节奏）

当你要继续写下一个组件（比如 `Alert` 或 `Input`），建议你保持同样的节奏：

- **先补变量**：把该组件需要的 tokens 放进 `styles/_variables.scss`
- **再补 mixin（可选）**：如果发现样式规律会被复用，就加到 `styles/_mixin.scss`
- **写 types**：先规定 props 和枚举
- **写 tsx**：只做语义结构 + class 拼接 + 少量交互
- **写 _style.scss**：用 variables + mixin 产出各类 class（size/type/state）
- **在 `styles/index.scss` 打开 import**：把该组件样式接入全局入口
- **在 `App.tsx` 写 demo 用例**：用不同 props 把边界情况跑一遍

## 你当前项目里“已经形成的规范”（你可以继续沿用）

- **样式文件命名**：`styles/_xxx.scss` 作为全局模块，组件内用 `_style.scss`
- **入口收口**：只在 `App.tsx` 里引 `styles/index.scss`（避免到处引样式导致顺序不确定）
- **class 命名模式**：`btn` + `btn-${type}` + `btn-${size}`（非常经典，也利于扩展）
