Bee-Design 项目架构说明（小白版）

一、图片里说的 Monorepo 是什么？
Monorepo（单仓多包）= “一个 Git 仓库里放多个彼此独立但可协作的包/子项目”。
常见长相：
- 根目录有 pnpm-workspace.yaml / lerna.json / turbo.json / nx.json 之类的配置
- 下面有 packages/xxx、apps/xxx 等多个 package.json
- 每个包都能单独发布/单独构建，但又能在同一个仓库里共享依赖、共享工具链

它和你平时常见的“一个仓库一个项目”不一样：
- 单仓单包（Single-repo Single-package）：1 个仓库里主要就 1 个可发布的 npm 包（或 1 个应用）
- 单仓多包（Monorepo）：1 个仓库里有 N 个包/应用，彼此引用与协作更紧密


二、我的 Bee-Design 是 Monorepo 吗？
结论：不是典型 Monorepo，而是“单仓单包”的组件库工程。

我在你的项目里看到的关键信号：
- 只有一个顶层 package.json，包名是 @aozi6666/bee-design
- 使用的是 npm 的 package-lock.json（而不是 pnpm workspace 的 pnpm-workspace.yaml）
- 项目结构以 src/ 为源码入口，rollup/ 为打包配置，.storybook/ 为组件文档/演示
- dist/ 是构建产物（按 ESM/CJS/UMD 多格式输出）

换句话说：你现在是把“组件库源码 + 打包/测试/文档工具”都放在同一个包工程里维护。


三、Bee-Design 真实采用的架构是什么？
可以理解为：组件库工程化架构（单仓单包）= “源码（src） + 构建（Rollup） + 开发调试（Vite） + 文档/演示（Storybook） + 测试（Jest） + 质量保障（ESLint/Prettier/Stylelint + Husky）”

1）源码层（你写组件的地方）
- src/
  - 组件源码、样式、入口导出等

2）开发调试层（本地开发体验）
- Vite（vite.config.ts）
  - 提供 dev server：npm run dev
  - 让你在本地快速跑起来、热更新、调试组件

3）构建发布层（把源码变成别人能安装使用的包）
- Rollup（rollup/ 目录下多份配置）
  - 打包输出多种模块格式：ESM / CJS / UMD
  - 对应 package.json 里的字段：
    - module -> dist/index.esm.js
    - main   -> dist/index.cjs
    - unpkg  -> dist/index.umd.js
  - types -> dist/index.d.ts（TypeScript 类型声明）

4）组件文档/演示层（给人看的“组件站点/故事集”）
- Storybook（.storybook/ + scripts）
  - storybook：本地预览与展示组件
  - build-storybook：生成静态站点（可部署）

5）测试层（保证组件行为稳定）
- Jest（scripts 里 test/test:ci + jest 配置在 package.json 内）

6）质量与规范层（避免越写越乱）
- ESLint：代码规范与错误检查
- Prettier：统一格式化
- Stylelint：CSS/SCSS 规范
- Husky + lint-staged：提交前自动跑校验，减少“坏代码进仓库”


四、这种架构的优势是什么？（为什么适合做组件库）
1）发布友好：一次构建，输出多种模块格式
- 你的组件库既能被现代打包工具按 ESM 引用，也能兼容需要 CJS 的场景
- UMD 还能给一些 “直接用 script 引入” 或 CDN 场景使用

2）开发效率高：Vite 开发很快
- 本地调试快、热更新快，写组件体验更顺滑

3）展示与协作更清晰：Storybook 把组件“可视化”
- 组件的用法、状态、交互可以被直接看到
- 适合给别人演示，也适合作为“组件说明书”

4）质量更稳定：测试 + 提交前校验
- Jest 能在你改动后帮你发现行为回归
- Husky 能把问题拦在提交前，减少线上/发布后翻车

5）上手门槛相对低（对小白更友好）
- 不需要先理解 workspace、包之间依赖、版本联动、发布流水线等 Monorepo 复杂度
- 先把“组件怎么写、怎么打包、怎么发布”跑通，更容易形成正反馈


五、那我什么时候才需要 Monorepo？
当你出现这些需求时，再考虑升级成 Monorepo 会更合适：
- 你想把“组件库 + 文档站点 + 工具库（utils）+ 脚手架”拆成多个包分别维护
- 你希望 docs-site 依赖 components 以“源码联动”的方式开发，而不是拷贝或相对路径 hack
- 你需要多个包独立发布（版本策略、变更日志、CI 流水线）并且团队协作变多

如果你还处在“先把组件库打磨好”的阶段：你现在这种单仓单包架构非常合理。
